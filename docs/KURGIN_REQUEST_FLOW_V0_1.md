# KURGIN Request Flow v0.1

## 1. Назначение

Request Flow v0.1 нужен, чтобы пользователь мог связаться с KURGIN по конкретному камню.

Это безопасный public contact-flow для камней в состоянии `request_price` / no-price. Он помогает пользователю отправить намерение связаться с менеджером, но не создаёт заказ, резерв, оплату или юридически подтверждённую заявку в backend.

## 2. Каналы связи

Текущий порядок каналов:

1. **MAX** — основной канал.
2. **Telegram** — второй канал.
3. **WhatsApp** — третий канал.

Каналы используются только как способ связи с KURGIN. Они не являются checkout, CRM, backend-заявкой или системой бронирования.

## 3. Где показывается

Request Flow v0.1 показывается:

- в `detail sheet` конкретного камня;
- для камней в состоянии `request_price` / no-price;
- через кнопку **«Запросить цену»**;
- через контактные кнопки **MAX / Telegram / WhatsApp**.

Основной сценарий:

```text
Камень без публичной цены
→ detail sheet
→ Запросить цену
→ MAX / Telegram / WhatsApp
```

## 4. Что request-flow НЕ означает

Request Flow v0.1 **не означает**:

- заказ;
- резерв;
- оплату;
- фиксацию цены;
- checkout;
- sold;
- ownership transfer;
- price lock;
- подтверждённую заявку в backend.

Переход в MAX / Telegram / WhatsApp — это только контактное действие пользователя.

## 5. Что сейчас НЕ делается

В текущей версии **не делается**:

- запись в admin;
- создание `requests.csv`;
- backend/API;
- CRM;
- автоматическое уведомление менеджера через систему;
- изменение `catalog.json`;
- изменение `kurgin-data`;
- изменение статуса камня;
- резервирование камня;
- подтверждение цены;
- перевод камня в `sold`.

## 6. Safe public wording

Разрешённый публичный текст:

> Заявка не является заказом, резервом, оплатой или фиксацией цены.

Допустимое расширение:

> Менеджер уточнит наличие и условия.

Запрещено формулировать этот flow как покупку, бронирование, оплату, фиксацию цены или юридически подтверждённый заказ.

## 7. Текущий статус

```text
Request Flow v0.1 = visual/contact flow only.
```

То есть текущая версия является только визуальным и контактным слоем публичного сайта.

Она не создаёт рабочую сущность заявки внутри KURGIN Admin, не пишет данные в backend и не меняет состояние камня.

## 8. Future upgrade

В будущем можно добавить полноценную систему заявок:

- `requests.csv` в админке;
- `request_id`;
- `request_type`;
- `stone_id`;
- `stone_snapshot`;
- контакты пользователя;
- статусы заявки:
  - `new`;
  - `in_review`;
  - `contacted`;
  - `closed`;
  - `cancelled`;
- admin requests panel;
- backend/API;
- CRM/webhook.

## 9. Что нельзя делать сейчас

В рамках Request Flow v0.1 нельзя:

- делать checkout;
- делать оплату;
- делать реальный резерв;
- переводить камень в `sold`;
- фиксировать цену;
- считать сообщение в мессенджере юридически подтверждённым заказом;
- считать переход в мессенджер созданной backend-заявкой;
- менять статус камня;
- записывать персональные данные без отдельного backend/API и политики обработки данных.

## 10. Контрольное правило

Любое расширение request-flow должно сохранять принцип:

```text
Contact intent first. Transaction later.
```

Сначала пользователь связывается с KURGIN. Любые заказ, резерв, оплата, price lock или изменение статуса камня требуют отдельного, явно утверждённого workflow.
